# T1.4 Implementation Summary: Lobby → GameLoop Strategy Handoff

## Objective
Establish data handoff protocol to pass selected strategies from LobbyScene to GameLoopScene.

## Phase 1 Scope (Current)
- **PRIMARY GOAL**: Pass strategies data from Lobby to GameLoop
- **SECONDARY**: Maintain SceneManager compatibility (CoreGameState passed through but not used)
- **NOT IN SCOPE**: Building game, creating game state, rendering game UI

## Implementation Details

### 1. LobbyScene (`scenes/lobby_scene.py`)
**Change**: Modified transition target from `"shop"` to `"game_loop"`

```python
# OLD:
self.scene_manager.request_transition("shop", strategies=list(self.ui_state.strategies))

# NEW:
self.scene_manager.request_transition("game_loop", strategies=list(self.ui_state.strategies))
```

**Purpose**: Route strategy selection to GameLoopScene instead of directly to ShopScene.

---

### 2. GameLoopScene (`scenes/game_loop_scene.py`) - NEW FILE
**Status**: Minimal implementation for Phase 1

**Constructor Signature**:
```python
def __init__(self, core_game_state, strategies: Optional[List[str]] = None)
```

**Key Attributes**:
- `self.strategies` - PRIMARY: Stores strategies from lobby
- `self.core_game_state` - SECONDARY: For SceneManager compatibility (not used in Phase 1)
- `self.scene_manager` - Set by SceneManager for transitions

**Methods Implemented**:
- `on_enter()` - Creates minimal UIState
- `on_exit()` - Discards UIState
- `handle_input(input_state)` - ESC key returns to lobby
- `update(dt)` - Empty (no logic yet)
- `draw(screen)` - Shows received strategies for validation

**Phase 1 Behavior**:
- Displays received strategies on screen
- Logs strategies to console
- ESC key returns to lobby
- No game logic, no turn management, no rendering beyond validation

2. Register `create_game_loop` factory

**Factory Implementation**:
```python
def create_game_loop(core_game_state, **kwargs):
    strategies = kwargs.get('strategies', None)
    print(f"[main.py] create_game_loop called with strategies: {strategies}")
    return GameLoopScene(core_game_state, strategies=strategies)
```

**Purpose**: Extract strategies from kwargs and pass to GameLoopScene constructor.

## Data Flow

```
User selects strategies in Lobby
    ↓
LobbyScene.handle_input() detects ENTER/START
    ↓
scene_manager.request_transition("game_loop", strategies=[...])
    ↓
SceneManager._execute_transition()
    ↓
create_game_loop(core_game_state, strategies=[...])
    ↓
GameLoopScene(core_game_state, strategies=[...])
    ↓
self.strategies = strategies  ← DATA RECEIVED
```

## Key Design Decisions

### Why CoreGameState is passed but not used?
SceneManager requires all scenes to have `core_game_state` attribute for:
1. Passing to next scene factory
2. Verifying same instance (no copying)

Phase 1 focuses on strategies data handoff, so CoreGameState is passed through but not actively used.

### Why not extend Scene base class?
GameLoopScene doesn't extend Scene because:
1. Phase 1 is minimal - just data handoff
2. Full Scene integration happens in T1.5
3. Keeps implementation simple and focused

### Why minimal rendering?
Phase 1 goal is data validation, not UI:
- Shows strategies were received correctly
- Provides visual confirmation for testing
- Full UI comes in later tasks
